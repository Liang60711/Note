## 描述 Bean 生命週期

1. 實例化 Bean : 
    - 使用反射或Factory方法創建。

2. 填充 Bean 屬性、依賴注入 : 
    - populateBean() 注入依賴。

    - 循環依賴由 Spring 的 三級緩存（singletonObjects、earlySingletonObjects、singletonFactories）解決。

3. 調用 aware 接口相關的方法 : 
    - 包括 BeanNameAware.setBeanName()、BeanFactoryAware.setBeanFactory()、ApplicationContextAware.setApplicationContext()。

    - 發生在屬性注入完成之後，初始化前。

4. 調用 BeanPostProcessor 前置處理方法 : 
    - postProcessBeforeInitialization()。

    - 可以對 Bean 進行修改或包裝。

5. 初始化 (Initialization) : 

    - 執行順序：

        1. @PostConstruct 方法

        2. InitializingBean.afterPropertiesSet() 方法

        3. @Bean(initMethod="...")（如果有設定）


6. 調用 BeanPostProcessor 後置處理。

    - postProcessAfterInitialization()

    - 常用於 AOP 或代理包裝。

7. Bean 就緒，可被使用

8. Spring 容器關閉時，Bean 被銷毀。

    - @PreDestroy 方法

    - DisposableBean.destroy()

    - @Bean(destroyMethod="...")（如果有設定）

## Jwt Token 驗證流程 (OAuth2.0 - `Password Grant` 授權方式)

限在內部系統使用的 SSO。

使用 Auth Server 和 Redis 作為驗證伺服器，Redis 只儲存 Refresh Token (Redis 不一定要使用，可以直接將 Refresh Token 存在資料庫)。

1. Client 使用帳號密碼在 Auth Server 成功登入。

2. Client 拿到 Access Token 後，直接帶入訪問後端，透過後端解析，即完成驗證(不用再去 Auth Server 驗證)。

3. 若 Access token 過期，則後端返回過期，並叫 Client 重新使用 Refresh Token 呼叫 Auth Server。

4. 此時 Auth Server 驗證 Refresh Token 的正確性，若無誤則重新發 Access Token和新的 Refresh Token 給Client，並刷新 Redis 中的 refresh token。

## OAuth2.0 / OIDC

以微服務舉例，以下為核心組件

- Authorization Server: 負責驗證帳密並發行 Token（例如：Spring Authorization Server）。
- API Gateway: 負責統一攔截請求、校驗 Token 合法性，或將驗證資訊轉發給後端。
- Resource Services (Microservices)：後端各個業務模組（如 Order Service, User Service），作為資源伺服器。
- Redis: 儲存 Refresh Token 或 Token 黑名單（撤銷清單）。

---

步驟:

* 第一階段：獲取 Token (Login)

    1. 請求登入：Client 帶帳密向 Gateway 發起請求，Gateway 將其轉發至 Auth Server。

    2. 驗證與核發：Auth Server 驗證身分成功後，生成 JWT Access Token 與 Refresh Token。

    3. 持久化：Auth Server 將 Refresh Token 資訊存入 Redis（以便後續刷新或強制登出）。

    4. 返回結果：將雙 Token 回傳給 Client，Client 儲存在本地（如 LocalStorage 或 Secure Storage）。

* 第二階段：訪問微服務 (Request)

    5. 帶 Token 訪問：Client 請求業務 API（例如 /api/orders），並在 Header 帶上 Authorization: Bearer <Access_Token>。

    6. Gateway 預檢：

        * Gateway 截獲請求，解析 JWT 簽名是否合法、是否過期。

        * (選擇性) Gateway 從 JWT 中提取 user_id 或 authorities。

    7. 下游轉發 (Token Relay)：Gateway 驗證通過後，將請求轉發給後端 Order Service。通常會透過過濾器（GlobalFilter）將用户信息放入 Header 傳遞。

    8. 業務執行：Order Service 根據 Gateway 傳來的用戶資訊執行邏輯，並回傳結果。

* 第三階段：Token 過期與刷新 (Refresh)

    9. 過期攔截：當 Access Token 過期，Gateway 或後端返回 401 Unauthorized。

    10. 刷新請求：Client 帶上 Refresh Token 呼叫 Auth Server 的刷新接口。

    11. Redis 檢核：Auth Server 在 Redis 中確認該 Refresh Token 是否有效且未被撤銷。

    12. 滾動更新：驗證通過後，Auth Server 刪除 Redis 中舊的 Refresh Token，發行一組全新的 Access & Refresh Token 給 Client。

## CQRS

Command Query Responsibility Segregation（命令查詢責任分離）。
是一種軟體架構模式，用來將 修改資料的操作（Command） 與 讀取資料的操作（Query） 分開設計和處理。
